## 5. Node Version Recommendation

**Use Node `22.x` (LTS) on both Vercel and Render.** (Local build/validation was performed on Node **24.11.0**, which also works.)

- **Constraints:** Express 5 requires Node ≥ 18; Next 15.5 requires Node ≥ 18.18. Node 20 or 22 LTS comfortably satisfy both.
- **Recommendation rationale:** Node 22 is the current LTS — the most stable, longest-supported target for production. Vercel and Render both default to Node 22 for new Node services as of 2026, so no override is strictly required, but pinning is advised for reproducibility.
- Neither `package.json` declares an `engines` field and no `.nvmrc` exists. This is acceptable (platform defaults are fine). If you want to lock it explicitly, set the Node version in the **platform dashboard** (Vercel Project Settings / Render Environment) — this avoids editing `package.json` and keeps the change out of the code, consistent with the minimal-change mandate.

## 6. Environment Variable Checklist

### Frontend (Vercel) — all must be `NEXT_PUBLIC_*` (client-exposed)

| Variable | Required? | Production value | Notes |
|---|---|---|---|
| `NEXT_PUBLIC_API_BASE_URL` | **Yes** | `https://<your-backend>.onrender.com` | ⚠️ Set to the **Render backend URL**, NOT `http://localhost:8000`. The `localhost` value only lives in `.env.local.example` as a template. |
| `NEXT_PUBLIC_DATA_SOURCE` | Recommended | `live` | `live` uses backend (auto-falls back to mock on failure); `mock` forces offline demo. |
| `NEXT_PUBLIC_CESIUM_ION_TOKEN` | Optional | _(your Cesium ion token)_ | If omitted, globe falls back to bundled offline imagery — app still works. |

> `CESIUM_BASE_URL` is **not** a deployment variable — it is hard-set to `/cesium` inside `next.config.mjs` and served from `public/cesium` (copied by the `postinstall` script). No action needed.

### Backend (Render) — server-side only (never `NEXT_PUBLIC_`)

| Variable | Required? | Notes |
|---|---|---|
| `GEMINI_API_KEY` | Optional | AI Sky Narrator. Without it, narration falls back to a templated summary from real data. |
| `N2YO_API_KEY` | Optional | ISS next-pass window. Without it, ISS position/altitude/velocity still real; only the "next pass" event is omitted. |
| `PORT` | **Do NOT set** | Render injects it automatically; server falls back to `8000` locally. |

> All other backend data sources (NASA JPL Horizons, Open-Meteo, Open Notify, CelesTrak, USNO, BigDataCloud) are **keyless** — no config needed.

**Security verified:** Only `*.env.example` / `.env.local.example` templates are tracked in git. The real `backend/.env` and `frontend/.env.local` are git-ignored. No secrets are committed. No hardcoded `localhost`/dev URL exists anywhere in `frontend/src` or `backend/src`.

## 9. Expected Render Configuration (Backend)

| Setting | Value |
|---|---|
| **Root Directory** | `backend` ⚠️ critical — without it Render builds the whole monorepo and fails |
| **Runtime** | Node |
| **Build Command** | **`npm install --include=dev`** ⚠️ see caveat below |
| **Start Command** | `npm start` (runs `tsx src/server.ts`) |
| **Health Check Path** | `/api/health` |
| **Node Version** | 22.x |
| **Environment Variables** | `GEMINI_API_KEY`, `N2YO_API_KEY` (both optional). **Do NOT set `PORT`** — Render injects it. |

> **⚠️ Critical caveat — `--include=dev`:** The backend has **no compile step**; it runs TypeScript directly at runtime via `tsx`, which lives in `devDependencies`. Render sets `NODE_ENV=production`, which makes a plain `npm install` skip dev-dependencies — the app would then crash with `tsx: not found`. The `--include=dev` flag forces `tsx`/`typescript` to install so the Start Command can run. This matches the existing `deployment_guide.md` and is the intended design — **the backend `package.json` was deliberately left unchanged** rather than relocating `tsx` to `dependencies`, to avoid altering the project's documented deployment contract.

### Known, accepted residual (does NOT block deployment)

`npm audit` on the frontend reports **2 moderate** advisories for a **postcss `8.4.31` bundled *inside* Next.js** (`node_modules/next/node_modules/postcss`) — the "XSS via unescaped `</style>` in CSS stringify" issue. This is:

- **Not** the top-level `postcss` (which is the safe `8.5.15` used by the build).
- Present in **every** current Next.js release — the advisory's affected range includes Next 16 stable, so upgrading further does **not** resolve it.
- **Low real-world risk:** it only triggers when untrusted CSS is run through postcss's stringifier; Next does not expose this path to runtime user input.
- Only "fixable" via `npm audit fix --force`, which would force an out-of-range/breaking change without actually removing the bundled dependency.

Per the minimal-change / maximum-stability mandate, this transitive advisory is **documented and left as-is**. The deployment-blocking **critical/high** Next.js vulnerabilities (auth bypass, RCE, DoS) are fully resolved by the `15.5.19` upgrade.
